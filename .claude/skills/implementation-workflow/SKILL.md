# 実装フロースキル

## 概要

機能追加・改修・バグ修正などの実装タスクを、再現性のある手順で進めるためのスキル。

---

## 手順

### 1. 目的と完了条件の確認

- 何を実現する変更かを明確化する
- 受け入れ条件（期待挙動、除外範囲、制約）を整理する
- 不明点がある場合は実装前に確認する

### 2. 現状調査

- 対象コード、関連モジュール、呼び出し経路を把握する
- 既存の規約・設計方針・テスト方針を確認する
- 影響範囲（機能、データ、API、UI、運用）を見積もる

### 3. 実装方針の決定

- 変更アプローチを比較し、最小リスクの案を選ぶ
- 変更単位を小さく分割する（段階的に検証可能な粒度）
- 必要なテスト更新方針を先に決める
- 新規ライブラリを追加・更新する場合は、最新バージョンと公式ドキュメントを確認してから採用する（古い情報で実装しない）

### 4. ユーザー確認（必須）

- 以下をユーザーに共有する
  - 実装方針: 変更対象ファイル、修正箇所、採用するパターン
  - 影響範囲: 関連機能、既存テストへの影響、運用への波及
  - テスト方針: 追加・更新するテストの観点、unit-test スキルの適用有無
- 明示的な承認を得るまで実装を開始しない
- 追加要望や方針変更があれば、方針を更新して再確認する

### 5. テスト作成（バグ修正のみ・実装前）

- テスト作成は unit-test スキルに従って実施する
- バグ修正では、まず不具合を再現するテストを作成する
- 再現テストが修正前に失敗することを確認する（テストが不具合を正しく検出できることを担保するため）
- 再現条件が不明確な場合は、調査結果をもとにテスト観点を再整理する

### 6. 実装

- 既存パターンと一貫する形でコードを変更する
- コメントは **なぜ** を残すために書く。**何をしているか** はコード・命名で伝える
  - 次の箇所には積極的にコメントを付ける: 非自明な制約・前提、意図的に選んだトレードオフ、ひと目で意図を読み取れない条件分岐やマジックナンバー
  - コードと命名で意図が伝わる箇所には冗長なコメントを追加しない
- 関連ドキュメントがある場合は、実装内容と整合するよう更新する
- 想定通りに動かない・解決の糸口が見えない状態が続く場合は、既存知識で粘らず、公式ドキュメントや最新のベストプラクティスを Web などで調べ直す

### 7. テスト作成（バグ修正以外・実装後）

- テスト作成は unit-test スキルに従って実施する
- バグ修正以外では、実装後に変更仕様へ対応するテストを作成または更新する
- 正常系・異常系・境界値を含めて、必要な観点をテストに反映する
- テストだけで仕様意図が読み取れる粒度を目指す

### 8. 検証

- 変更箇所に対応するテストを実行する
- 必要に応じて静的解析・Lint・型チェックを実行する
- 想定したエッジケースと回帰影響を確認する

### 9. code-review サブエージェントへの依頼と対応

- コードレビューは `code-review` サブエージェント（`agents/code-review.agent.md`）に依頼する
- サブエージェントが返した指摘や懸念を反映して修正し、必要なテストを追加する
- 指摘対応後に再検証し、未解決事項は明確に残す

### 10. 仕上げ

- 差分を見直し、不要変更やノイズを除去する
- 変更内容と検証結果を要約する
- 次に必要な作業（追加テスト、段階リリース、監視項目）を整理する

## チェックリスト

### 要件
- [ ] 変更の目的と完了条件が明確
- [ ] スコープ外の作業が混入していない
- [ ] 実装方針についてユーザー承認を取得した

### 設計・実装
- [ ] 既存設計との整合性がある
- [ ] エラーハンドリングと境界条件を考慮している
- [ ] 変更単位がレビューしやすい粒度になっている
- [ ] 追加・更新するライブラリは最新バージョン・公式ドキュメントを確認した

### 検証
- [ ] 変更に対応するテストを追加・更新した
- [ ] テスト作成時に unit-test スキルを適用した
- [ ] バグ修正では、再現テストが修正前に失敗することを確認した
- [ ] バグ修正以外では、実装後にテストを追加・更新した
- [ ] 正常系・異常系・境界値をテストで確認した
- [ ] 主要な回帰経路を確認した
- [ ] 必要な静的チェックを通過した

### レビュー
- [ ] `code-review` サブエージェントにレビューを依頼した
- [ ] 指摘事項への対応と再検証が完了している

### 共有
- [ ] 変更内容を簡潔に説明できる
- [ ] 必要なドキュメント更新が反映されている

---

## 行動ルール

- **まず理解してから変更する** — 推測で書き始めない
- **小さく実装し、小さく検証する** — 一度に大きく変更しない
- **根拠を残す** — 重要な判断には理由を示す
- **既存規約を優先する** — 新規パターンは必要性がある場合のみ導入する
- **検証を省略しない** — 実装完了と検証完了を分けて扱う
- **最新情報を確認する** — ライブラリ追加時はバージョン・公式ドキュメントを確認し、実装が行き詰まったときは最新のベストプラクティスを調べる
- **このスキルも必要時に更新する** — 反復的に有効な改善点が確認できたら `SKILL.md` を更新し、プロジェクトに即したものに改善していく
