# コミットスキル

## 概要

変更内容を確認したうえで、Conventional Commits 形式のメッセージでコミットするためのスキル。

---

## 手順

### 1. 変更内容の確認

- `git status` で対象ファイルとステージ状態を確認する
- `git diff` と `git diff --staged` で差分を確認する
- 意図しない変更やノイズが混ざっていないことを確認する

### 2. コミット単位の整理

- 1コミット1目的になるように変更を分割する
- 目的が異なる変更は別コミットに分離する
- 1コミット1目的の範囲に該当するファイルのみを `git add <path>` で個別にステージする（`git add -A` や `git add .` のような一括追加は避ける）

### 3. 過去コミットメッセージの参照

- `git log --oneline -n 20` などで直近のコミット履歴を確認し、リポジトリで定着している傾向を把握する
- 件名の言語（日本語／英語）、`type(scope)` の使い方、`scope` の命名、句読点や語尾の癖など、揃えるべき様式を拾う
- 以降の手順では、原則としてこの様式に合わせる（Conventional Commits の範囲で逸脱しない限り）

### 4. Conventional Commits メッセージ作成

メッセージは次の構造で作成する。

```text
<type>(scope): <description>

<body>
```

- `scope` は必要な場合のみ付与する
- `body` は必要な場合のみ記載する

`type` は以下から選ぶ。

- `feat`: ユーザー向け機能追加
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: 振る舞いに影響しない整形やスタイル変更
- `refactor`: 挙動を変えないリファクタリング
- `perf`: パフォーマンス改善
- `test`: テスト追加・修正
- `build`: ビルドシステムや依存管理の変更
- `ci`: CI 設定やスクリプトの変更
- `chore`: その他の雑務的変更
- `revert`: 既存コミットの取り消し

### 5. 件名・本文のルール

- 件名は簡潔に現在形で書く
- 件名の先頭文字は小文字を基本とする
- 件名末尾に句点を付けない
- 本文は必要な場合のみ追加し、変更理由を簡潔に補足する
- 過去コミットの様式（手順3）と矛盾する場合は、既存様式を優先する

### 6. ユーザー承認（必須）

- 共有する内容: コミット対象ファイル一覧、コミットメッセージ案（件名と本文）
- ユーザーから明示的な承認（「OK」「はい」「承認」など）を得るまで `git commit` を実行しない
- 修正依頼があれば反映し、再度同じ内容を共有して承認を取り直す

### 7. コミット実行

- `git commit -m "<subject>"` を実行する
- 本文が必要な場合は `-m` を追加して複数段で渡す
- 実行後に `git log --oneline -n 1` で結果を確認する

---

## メッセージ例

```text
feat: ログイン時のセッション更新処理を追加
```

```text
fix: 空文字入力時にエラーにならないよう修正

空文字の場合は早期 return し、添字アクセスを行わないようにする。
```

---

## チェックリスト

### 差分確認
- [ ] 意図したファイルだけがステージされている
- [ ] 無関係な変更やデバッグコードが含まれていない

### メッセージ品質
- [ ] Conventional Commits 形式に従っている
- [ ] `type` が変更内容と一致している
- [ ] 件名が簡潔で具体的
- [ ] 過去コミットの様式（言語、`scope` 命名、文体）と揃っている

### 実行前後
- [ ] コミット前にユーザー承認を取得した
- [ ] コミット後に直近履歴を確認した

---

## 行動ルール

- コミット前に必ず差分を読み直す
- 迷ったらコミットを分割し、1コミット1目的を優先する
- メッセージは変更理由が後から追える情報を残す
- 承認前に `git commit` を実行しない
- このスキルも必要時に更新する — 反復的に有効な改善点が確認できたら `SKILL.md` を更新し、プロジェクトに即したものに改善していく
